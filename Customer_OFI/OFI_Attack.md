# Checklist to include IEC 62443 in FEED documents for Shanghai CSCs

Including IEC 62443 in FEED for the Shanghai CSC means baking security into scope, architecture, and vendor specs rather than adding it at SAT. Use this as a **checklist you hand to the engineering / FEED lead**.

## Scope and objectives

- Define **IACS in scope** for the CSC: pilot lines, labs, test rigs, building automation that touch product or demos.[^8_1][^8_2]
- Add **cybersecurity posture** as an explicit FEED KPI alongside throughput, OEE, safety, and energy.[^8_3][^8_1]
- Decide **target Security Levels (SL)** per IEC 62443 (e.g., SL2 for labs, SL3 for any safety‚Äëcritical or customer-facing demo lines).[^8_4][^8_5]


## Risk assessment and CSRS

- Perform a **high‚Äëlevel cyber risk assessment** for the CSC as part of FEED (use IEC 62443‚Äë3‚Äë2 style approach).[^8_2][^8_6]
- Define the **System under Consideration (SuC)** and document external interfaces (corporate network, cloud, vendors).[^8_2]
- Create an initial **Cyber Security Requirements Specification (CSRS)** before major package procurement.[^8_7][^8_2]


## Zones, conduits, and architecture

- Produce a **zones and conduits diagram** for the CSC (engineering labs, OT network, DMZ, corporate IT, vendor access).[^8_8][^8_9]
- Assign **Security Level Targets** to each zone (SLT per IEC 62443‚Äë3‚Äë3) and record justification in FEED.[^8_5][^8_10]
- Specify **network segmentation** requirements:
    - Separate OT from IT; dedicated OT VLANs and firewalled conduits.[^8_9][^8_5]
    - DMZ for data exchange (historians, integration with SAP/Braincube, etc.).[^8_11][^8_5]
    - Remote/vendor access via jump hosts, not direct to controllers.[^8_6][^8_5]


## Technical requirements in FEED specs

- Identity \& access: unique accounts, RBAC, MFA for remote access, password policies for all OT systems.[^8_12][^8_5]
- System integrity: hardening baselines for PLCs, HMIs, servers; patch and configuration management expectations.[^8_13][^8_5]
- Data confidentiality and integrity: encryption for data crossing IT/OT and cloud interfaces; logging for changes to critical parameters.[^8_3][^8_11]
- Monitoring: requirement for **OT monitoring / logging** (syslog, NetFlow, or dedicated ICS security sensors) with export to SOC/SIEM.[^8_5][^8_3]


## Supplier and package requirements

- State in all **EPC and vendor RFQs** that OT systems must comply with relevant IEC 62443 parts (e.g., 4‚Äë1 for secure development, 4‚Äë2 for components).[^8_14][^8_13][^8_12]
- Require vendors to provide:
    - Secure development lifecycle evidence (SDL, patch policy, vulnerability disclosure).[^8_14][^8_13]
    - Hardening guides and recommended firewall rules for their equipment.[^8_13][^8_5]
    - Default‚Äëcredential removal and role-based access preconfigured.[^8_5]


## Testing, FAT/SAT, and commissioning

- Embed **cybersecurity test cases** into FAT/SAT scripts: account management, logging, network segregation verification.[^8_15][^8_5]
- Define **acceptance criteria**: no default passwords, no flat networks between IT and OT, all logging enabled as per CSRS.[^8_6][^8_5]
- Include a **cybersecurity punch‚Äëlist** in commissioning: open ports reviewed, firewall rules validated, remote access paths tested and documented.[^8_1][^8_15]


## Documentation and handover

- Ensure FEED deliverables include:
    - Zones \& conduits drawings, SL assignments, and CSRS.[^8_10][^8_6]
    - Updated OT/IT architecture diagrams with security devices identified.[^8_8][^8_9]
    - Cybersecurity O\&M guidance (patching, backup, incident response contacts).[^8_11][^8_5]
- Align documentation with **IEC 62443 and NIST manufacturing profile** language so it can be reused for audits and insurance.[^8_16][^8_13]

# List mandatory IEC 62443 clauses to cite in FEED documents

For FEED, you don‚Äôt need to quote every line of IEC 62443; you anchor the design to the key parts and concepts that drive architecture, vendor specs, and testing.

Below is a concise list of **‚Äúmust‚Äëcite‚Äù IEC 62443 parts/clauses** that should appear in Shanghai CSC FEED documents.

***

## Core standards to reference

Include these as normative references in the FEED ‚Äúcodes and standards‚Äù section:

- **IEC 62443‚Äë1‚Äë1 / ISA‚Äë62443‚Äë1‚Äë1** ‚Äì Concepts, models, terminology
    - Use to justify: zones \& conduits model, security levels, foundational requirements (FRs).[^9_1][^9_2]
- **IEC 62443‚Äë3‚Äë2** ‚Äì Risk assessment \& system partitioning
    - FEED language:
        - Define the *System under Consideration (SuC)*.
        - Partition SuC into **zones and conduits** and assign **Security Level Targets (SL‚ÄëT)** based on risk assessment.[^9_3][^9_2][^9_4]
- **IEC 62443‚Äë3‚Äë3** ‚Äì System security requirements \& security levels
    - FEED language:
        - The CSC IACS **shall meet IEC 62443‚Äë3‚Äë3 system requirements (SRs) aligned to SL‚ÄëT for each zone**.
        - Reference the **7 Foundational Requirements (FR1‚ÄìFR7)** as the baseline control set: Identification \& Authentication, Use Control, System Integrity, Data Confidentiality, Restricted Data Flow, Timely Response to Events, Resource Availability.[^9_5][^9_6][^9_3][^9_1]
- **IEC 62443‚Äë4‚Äë1** ‚Äì Secure development lifecycle (for vendors)
    - FEED language (in RFQs/specs for OEMs, integrators):
        - Suppliers of automation and control components **shall have a secure product development lifecycle compliant with IEC 62443‚Äë4‚Äë1** (requirements covering requirements management, design, implementation, verification/validation, patch and vulnerability management, and product end‚Äëof‚Äëlife).[^9_6][^9_7][^9_8]
- **IEC 62443‚Äë4‚Äë2** ‚Äì Component technical security requirements
    - FEED language:
        - PLCs, HMIs, network and host devices **shall be capable of meeting IEC 62443‚Äë4‚Äë2 Component Requirements (CRs)** at the SL required for their zone (e.g., SL 2 or SL 3).
        - Vendors must declare supported **Security Level‚ÄëCapability (SL‚ÄëC)** per device.[^9_9][^9_10][^9_8][^9_3]

***

## Mandatory design concepts to call out

These aren‚Äôt ‚Äúclauses‚Äù by number, but concepts FEED must treat as requirements:

### 1. Zones and conduits (3‚Äë2 / 3‚Äë3)

- FEED shall deliver:
    - A **zones and conduits diagram** for Shanghai CSC OT/IT, aligned with IEC 62443‚Äë3‚Äë2.[^9_2][^9_4][^9_3][^9_6]
    - Assigned **SL‚ÄëT per zone** with rationale in the Cybersecurity Requirements Specification (CSRS).[^9_11][^9_3]


### 2. Security levels (SL‚ÄëT, SL‚ÄëC, SL‚ÄëA)

- FEED shall:
    - Define **SL‚ÄëT** for each zone/conduit (e.g., SL 2 for labs, SL 3 for safety‚Äëcritical/demo lines).[^9_11][^9_9][^9_1]
    - Require components with appropriate **SL‚ÄëC** (per 3‚Äë3 / 4‚Äë2) and design to achieve **SL‚ÄëA** (achieved level).[^9_9][^9_11]


### 3. Foundational Requirements (FR1‚ÄìFR7) mapped to SRs

In the system requirements section of FEED, explicitly state that the design must implement IEC 62443‚Äë3‚Äë3 SRs aligned to these FRs:[^9_3][^9_1][^9_5][^9_6]

- **FR1 ‚Äì Identification \& Authentication Control (IAC/AC)**
- **FR2 ‚Äì Use Control (UC)**
- **FR3 ‚Äì System Integrity (SI)**
- **FR4 ‚Äì Data Confidentiality (DC)**
- **FR5 ‚Äì Restricted Data Flow (RDF)**
- **FR6 ‚Äì Timely Response to Events (TRE)**
- **FR7 ‚Äì Resource Availability (RA)**

# How to state SL TM and SL C targets in FEED

In FEED, SL‚ÄëT and SL‚ÄëC should be stated **formally and unambiguously** so engineers, suppliers, and OT security all know the targets and what must be delivered. Use simple, repeatable wording tied to IEC 62443 terms.

***

## 1. Concepts to reflect in FEED

- **SL‚ÄëT (Target Security Level):**
The **risk‚Äëbased security goal** for each zone/conduit in the system, defined by the asset owner using IEC 62443‚Äë3‚Äë2 and expressed on the SL 0‚Äì4 scale.[^10_1][^10_2][^10_3]
- **SL‚ÄëC (Capability Security Level):**
The maximum security level each **system or component can natively support**, based on IEC 62443‚Äë3‚Äë3 (system SRs) and 62443‚Äë4‚Äë2 (component CRs).[^10_4][^10_5][^10_1]
- Relationship you want FEED to enforce:
**SL‚ÄëC ‚â• SL‚ÄëT**, and at commissioning **SL‚ÄëA ‚â• SL‚ÄëT** for each zone/conduit.[^10_6][^10_7][^10_4]

***

## 2. Template wording for SL‚ÄëT in FEED

Use this structure in the **Cybersecurity Requirements / Architecture** section:

> ‚ÄúSecurity Levels shall be defined in accordance with **IEC 62443‚Äë3‚Äë2 and IEC 62443‚Äë3‚Äë3**. For each security zone and conduit in the Shanghai CSC System under Consideration (SuC), a **Target Security Level (SL‚ÄëT)** on the 0‚Äì4 scale shall be assigned based on the cyber risk assessment.‚Äù

Then define per‚Äëzone targets in a table, for example:


| Zone / Conduit | Description | SL‚ÄëT | Rationale |
| :-- | :-- | :-- | :-- |
| Z1 ‚Äì Demo / Pilot Line OT | Customer‚Äëfacing demo line PLCs, HMIs, safety I/O | **SL‚ÄëT = 3** | Must withstand deliberate, skilled attacks with moderate resources. [^10_2][^10_8] |
| Z2 ‚Äì Lab / Application Dev | Non‚Äëproduction test rigs, dev systems | **SL‚ÄëT = 2** | Protection against intentional misuse with basic tools. [^10_2][^10_3] |
| Z3 ‚Äì OT‚ÄìIT DMZ | Historians, integration servers, jump hosts | **SL‚ÄëT = 2** | Conduit between corporate IT and OT zones. [^10_3][^10_9] |
| Z4 ‚Äì Building Automation | HVAC, lighting, non‚Äëcritical BMS | **SL‚ÄëT = 1** | Protection against casual misuse. [^10_2] |

Boilerplate sentence to include under the table:

> ‚ÄúThe **Cybersecurity Requirements Specification (CSRS)** shall record SL‚ÄëT values and justifications for all zones and conduits as required by IEC 62443‚Äë3‚Äë2.‚Äù[^10_3][^10_10]

***

## 3. Template wording for SL‚ÄëC in FEED

In the **Vendor / Package Requirements** section, specify SL‚ÄëC like this:

> ‚ÄúAll IACS components (PLCs, RTUs, HMIs, embedded devices, host systems, networking and security devices) supplied for the Shanghai CSC shall demonstrate **Capability Security Levels (SL‚ÄëC)** in accordance with **IEC 62443‚Äë3‚Äë3 and IEC 62443‚Äë4‚Äë2**. For each zone, component SL‚ÄëC **shall be equal to or greater than the SL‚ÄëT** defined for that zone.‚Äù[^10_5][^10_11][^10_1][^10_4]

Then specify what information vendors must provide:

- ‚ÄúEach vendor **shall provide an SL‚ÄëC declaration** for their product, including:
    - Mapping of the product‚Äôs security functions to relevant **System Requirements (SR)** or **Component Requirements (CR)** and Requirement Enhancements (RE) from IEC 62443‚Äë3‚Äë3 / 62443‚Äë4‚Äë2.[^10_12][^10_1]
    - The maximum SL‚ÄëC (0‚Äì4) for each FR category (FR1‚ÄìFR7).[^10_12][^10_5]
    - Any **compensating countermeasures** needed if SL‚ÄëC < SL‚ÄëT for a given FR, so the integrator can design to meet SL‚ÄëT. ‚Äù[^10_13][^10_1][^10_6]

You can also add a simple compliance rule:

> ‚ÄúFor all zones in the CSC SuC, **SL‚ÄëC (component) ‚â• SL‚ÄëT (zone)** shall be used as the minimum selection criterion. Where this is not feasible, the FEED shall identify and cost **compensating controls** needed to achieve SL‚ÄëT.‚Äù[^10_7][^10_6][^10_4]

***

## 4. Short, ‚Äúcopy‚Äëpaste‚Äù examples

Use these exact sentences in FEED:

- **Global statement:**
    - ‚ÄúSecurity Levels shall follow IEC 62443 conventions: **SL‚ÄëT** (Target) set by risk assessment, **SL‚ÄëC** (Capability) declared by suppliers, and **SL‚ÄëA** (Achieved) verified during acceptance testing. - **Per‚Äëzone requirement:**
    - ‚ÄúZone Z1 ‚Äì Demo / Pilot OT shall achieve **SL‚ÄëT = 3** for FR1‚ÄìFR7. All PLCs, HMIs, and network devices within Z1 shall have **SL‚ÄëC ‚â• 3** for the same foundational requirements or be supplemented with compensating controls defined in the CSRS.‚Äù[^10_2][^10_7][^10_12]
- **Vendor requirement:**
    - ‚ÄúThe supplier shall provide documentation demonstrating that the proposed component meets **IEC 62443‚Äë4‚Äë2** and supports **SL‚ÄëC = 2** (minimum) for Identification \& Authentication, Use Control, System Integrity, and Restricted Data Flow.‚Äù[^10_11][^10_5][^10_12]
